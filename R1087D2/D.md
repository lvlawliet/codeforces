### 1. 题目翻译

**题目：D. Ghostfires (鬼火)**

**题目描述：**
OtterZ 决定用鬼火制作一把武器。他收集了一些红色（Red）、绿色（Green）和蓝色（Blue）的鬼火。他需要将这些鬼火排成一排来组装武器。为了让武器威力最大化，他希望使用**尽可能多**的鬼火。但是，如果排列中出现以下情况，武器就会失控：
* 相邻的两个鬼火颜色相同。
* 两个颜色相同的鬼火之间刚好隔着两个其他的鬼火（即相距为 3 的两个鬼火颜色相同）。

形式化地说，OtterZ 有 $r$ 个红色、 $g$ 个绿色、 $b$ 个蓝色的鬼火。他想构造一个由字符 'R', 'G', 'B' 组成的字符串 $s$，满足以下条件：
1. 'R', 'G', 'B' 在字符串中的出现次数分别不超过 $r, g, b$。
2. 对于所有 $1 \le i \le |s|-1$，满足 $s_i \neq s_{i+1}$（相邻不同）。
3. 对于所有 $1 \le i \le |s|-3$，满足 $s_i \neq s_{i+3}$（相隔 2 个不同）。
4. 字符串 $s$ 的长度**尽可能长**。

请帮助 OtterZ 构造出这个序列。如果有多种可能的最长序列，输出任意一种即可。

**输入格式：**
* 第一行是测试用例数量 $t$ ($1 \le t \le 10^4$)。
* 每个测试用例只有一行，包含三个整数 $r, g, b$ ($0 \le r, g, b \le 10^6$ 且 $r+g+b > 0$)。
* 保证所有测试用例的 $r+g+b$ 之和不超过 $10^6$。

**输出格式：**
* 输出由 'R', 'G', 'B' 构成的字符串，代表鬼火的颜色排列。多解时输出任意一个。

---

### 2. 解题思路

**算法剖析：**

这是一道典型的**贪心构造题**。我们要逐个确定字符串 $s$ 每一位上的字符。

为了让字符串尽可能长，我们要尽量避免提前把某一种颜色消耗光而导致后面无路可走（也就是“死胡同”）。因此，核心的贪心策略是：**在满足放置条件的前提下，优先放置剩余数量最多的那一种颜色的鬼火。**

具体到每一步（假设当前要决定第 $n$ 个字符，即 $s[n]$）：
1. **合法性检查**：我们要选的颜色，不能和 $s[n-1]$ 相同（规则一），也不能和 $s[n-3]$ 相同（规则二）。
2. **贪心选择**：在所有合法的颜色中，选择**剩余数量最大**（`counts[c] > max_cnt`）的颜色。
3. **巧妙的平局打破（Tie-breaker）**：如果有两种或多种合法颜色的剩余数量一样多，该选哪一个？代码中给出了一个非常聪明的优先策略：**优先选择与 $s[n-2]$ 颜色相同的那个**。
   * **为什么？** 因为规则是 $s_i \neq s_{i+1}$ 且 $s_i \neq s_{i+3}$。如果我们让 $s_n = s_{n-2}$（即形成 `A B A` 这样的局部结构），那么后续我们在选择 $s_{n+1}$ 时，它不能等于 $s_n$（即 `A`），也不能等于 $s_{n-2}$（还是 `A`）。这意味着我们用相同的颜色覆盖了两个限制槽位，从而为 $s_{n+1}$ 保留了更多的颜色选择余地（另一种颜色完全没有被限制）。这种选择方式更容易形成 `R G R B R G R B...` 这种高密度的稳定循环。

如果某一步找不到任何合法的颜色，说明无法继续构造，直接结束当前字符串的构造并输出。

**复杂度分析：**
* **时间复杂度**：每次构造一个字符，最多需要遍历 3 种颜色进行常数级别的判断。因此构造单个测试用例的时间复杂度为 $O(|s|)$，其中 $|s| \le r+g+b$。总时间复杂度为 $O(\sum (r+g+b))$，由于题目保证总和不超过 $10^6$，可以在 2 秒内极速跑完。
* **空间复杂度**： $O(\sum(r+g+b))$，用于存储答案字符串 `s`。使用了 `s.reserve()` 预分配内存，避免了动态扩容带来的开销，非常高效。

### 3. 代码详解

```cpp
#include <iostream>
#include <string>
#include <vector>

using namespace std;

void solve()
{
    // counts 数组存储 R, G, B 的剩余数量
    int counts[3];
    cin >> counts[0] >> counts[1] >> counts[2];
    char colors[3] = {'R', 'G', 'B'};

    string s = "";
    // 提前分配最大可能需要的内存，防止字符串拼接时不断重新分配内存导致的耗时
    s.reserve(counts[0] + counts[1] + counts[2]);

    while (true)
    {
        int best_c = -1;       // 记录当前这一步最佳颜色的索引 (0, 1 或 2)
        int max_cnt = -1;      // 记录候选颜色中的最大剩余数量
        int n = s.length();    // 当前已经构造的字符串长度

        // 遍历三种颜色，寻找最合适的选择
        for (int c = 0; c < 3; ++c)
        {
            // 条件 1：这种颜色必须还有剩余
            if (counts[c] == 0)
                continue;
            // 条件 2：不能和前一个颜色相邻 (即 s[n] != s[n-1])
            if (n >= 1 && colors[c] == s[n - 1])
                continue;
            // 条件 3：不能和前三个颜色相同 (即 s[n] != s[n-3])
            if (n >= 3 && colors[c] == s[n - 3])
                continue;

            // 判断当前颜色是否与 s[n-2] 相同，如果相同则标记为 prefer (偏好)
            bool prefer = false;
            if (n >= 2 && colors[c] == s[n - 2])
            {
                prefer = true;
            }

            // 贪心策略：优先选择剩余数量更多的颜色
            if (counts[c] > max_cnt)
            {
                max_cnt = counts[c];
                best_c = c;
            }
            // 平局打破策略：如果剩余数量一样多，优先选择等于 s[n-2] 的颜色
            else if (counts[c] == max_cnt)
            {
                if (prefer)
                {
                    best_c = c;
                }
            }
        }

        // 如果遍历完 3 种颜色都没有找到合法的，说明无法再往后加了，跳出循环
        if (best_c == -1)
        {
            break;
        }

        // 将选定好的最佳颜色追加到字符串中
        s += colors[best_c];
        // 扣除消耗掉的鬼火数量
        counts[best_c]--;
    }

    // 输出最终构造的最长字符串
    cout << s << "\n";
}

int main()
{
    // C++ 经典的 I/O 竞程优化，解除 cin/cout 和 stdio 的同步，极大地提升读写速度
    ios_base::sync_with_stdio(false);
    cin.tie(NULL);

    int t;
    // 不断读取测试用例数量 t，直到没有输入为止
    if (cin >> t)
    {
        while (t--)
        {
            solve();
        }
    }
    return 0;
}
```
